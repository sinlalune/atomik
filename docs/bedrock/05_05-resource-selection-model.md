# Resource, view, selection, and scope model

## Why this matters

The first MVP can stay simple while preserving the future if it uses stable abstractions.

```text
Project = durable working directory for a subject
Resource = thing opened by Atomik
View = how the resource is displayed
Selection = what the user focuses on
Scope = where the agent is allowed or expected to look
AI operation = transformation applied to selections and scope
Artifact = output generated by AI
Patch = proposed file change
```

## Type sketch

```ts
type Resource =
  | ProjectResource
  | NoteResource
  | SourceDossierResource
  | WebResource
  | PdfResource
  | ImageResource
  | AudioResource
  | CodeResource
  | TranscriptResource
  | FutureSceneResource
  | FutureCanvasResource

type ViewKind =
  | 'project-overview'
  | 'markdown-editor'
  | 'markdown-preview'
  | 'source-dossier'
  | 'source-image'
  | 'source-audio'
  | 'source-pdf'
  | 'source-web'
  | 'ai-panel'
  | 'agent-context'
  | 'dev-docs'
  | 'git-diff'
  | 'operation-receipt'
  | 'future-scene'
  | 'future-canvas'

type Selection = {
  resourceId: string
  viewId: string
  kind: 'text' | 'image-region' | 'code' | 'math' | 'block' | 'page' | 'time-range' | 'file'
  content?: string
  anchor?: SourceAnchor
  metadata?: Record<string, unknown>
}

type ContextScope =
  | { kind: 'vault'; root: string }
  | { kind: 'project'; root: string }
  | { kind: 'folder'; path: string }
  | { kind: 'source'; dossierPath: string }
  | { kind: 'note'; path: string }
  | { kind: 'selection'; selection: Selection }
```

## Core invariant

The UI should not hand random component state to AI. It should hand normalized `Selection` and `ContextScope` objects.

```text
selection = what the user is pointing at
scope = where the agent should look for surrounding knowledge
```

## Scope examples

```text
Ask about this highlighted PDF paragraph
  selection: PDF text range
  scope: source dossier folder

Ask how this note fits the project
  selection: note file
  scope: project root

Ask for a roadmap update
  selection: current roadmap section
  scope: docs/bedrock + docs/adr

Ask for a global cleanup search
  selection: none
  scope: vault root
```

## Anchor model

Source anchors may be simple or precise:

```ts
type SourceAnchor = {
  id: string
  label?: string
  target?: string
  page?: number
  textRange?: { start: number; end: number }
  imageRegion?: { x: number; y: number; width: number; height: number }
  timeRange?: { startSeconds: number; endSeconds: number }
  metadata?: Record<string, unknown>
}
```

Markdown should carry ordinary anchors and provenance wherever it is readable. Optional sidecars may carry coordinate-heavy details when needed.

## Future proofing

When Atomik DSL arrives, a DSL block is just another selectable block. When canvas arrives, a canvas node or edge is just another selectable resource fragment.

When deep agent context arrives, it should be a scoped traversal over files:

```text
scope -> index.md/log.md -> links/backlinks/tags -> selected files -> context budget
```

## Truth-aware extensions

Selection and claim are related but different:

```text
Selection
  what the user focused on in a resource or view

Claim
  an assertion introduced, extracted, or questioned

Evidence
  a source fragment used to support, contradict, qualify, define, or contextualize a claim

Verification request
  a bounded instruction to test one or more claims
```

```ts
type ClaimCandidate = {
  id: string
  statement: string
  sourceSelectionIds?: string[]
  nature?: string
  scope?: Record<string, unknown>
  introducedBy: 'user' | 'model' | 'source-extract' | 'mixed'
}

type EvidenceAnchor = {
  selection: Selection
  role: 'supports' | 'contradicts' | 'qualifies' | 'background' | 'defines'
  sourceClass?: 'primary' | 'secondary' | 'tertiary' | 'user-source' | 'unknown'
}
```

A text selection can contain several claims. One claim can be supported by several selections. Exact range mapping is optional for lightweight notes and required only when the user promotes, verifies, disputes, or cites the claim precisely.


## Scope, retrieval, and execution stay separate

```text
Selection       = what the user points at
ContextScope    = where Atomik may look
RetrievalPlan   = which direct/lexical/link/structural/semantic/external stages to use
ContextPacket   = the bounded inspectable material selected
ExecutionPolicy = deterministic/local/cloud/web permissions and privacy mode
OperationBudget = hard token/query/time/external-cost limits
ActionTrace     = what actually ran and what outcome followed
```

A selected paragraph can compile directly into context without vault search. A project-wide conceptual query may use hybrid retrieval. The same ContextPacket may be handled by a deterministic tool, local model, or cloud model under different policies without changing the canonical source files.
